Context: dLocal is already a payment provider in Mexico. What is missing is the dLocal wallet (saved cards on dLocal) running in parallel with the existing Stripe wallet.

In one sentence: Many patients in Mexico already pay with cards saved in the Stripe wallet. Enabling the dLocal wallet in a way that disables Stripe wallet would push those users toward high-friction checkout and risks lower online TPV, more in-office payment, and doctor-facing complaints.

What we want — two wallets running in parallel (combined /wallet UI, per-customer-provider behaviour) for two reasons:

  • Keep Stripe wallet alive — continue accepting payments from cards already saved there (recurrent visit pay-ins, TPV protection).
  • Enable the dLocal wallet roadmap — unlock planned experiments such as BNPL and recurrent payments using cards saved to dLocal.

Scope note: Running both wallets in parallel is a transitional state. Stripe wallet is preserved only for continuity of cards already saved there; the strategic direction for new wallet capabilities is dLocal. We will not dedupe cards across providers in /wallet — if the same card is added to both Stripe and dLocal, both entries will be shown. Cross-provider dedupe is out of scope — overkill for a non-permanent dual-wallet state.

Evidence snapshot (BroadDB, MX — directional)

  • ~68k patients with authorised Stripe wallet cards on file
  • ~23k captured saved-card marketplace pay-ins in 2026 (Stripe wallet)
  • ~2.6k patients with >1 visit booking on the same saved Stripe card in 2026
  • Monthly TPV in the table is per calendar month (not a YTD sum). Example: Mar 2026 alone ≈ 7.6k txs / ~7.3M MXN — see full table in the briefing link above
